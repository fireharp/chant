# US-007 — Verify failing marks the result NOT trusted

**As an** agent reusing a recipe whose verifier fails
**I want** `chant verify` to refuse to bless the result
**So that** a cache hit never becomes a wrong-answer amplifier.

## Acceptance criteria

- [ ] When the verifier command exits non-zero (or a declared expected artifact is missing), `chant verify --json <id>` **must** set `"trusted": false`.
- [ ] The outcome `message` **must** be `verifier did NOT pass — result is NOT trusted; repair or invalidate`.
- [ ] The outcome `recommended_next_command` **must** be `chant invalidate <id>`.
- [ ] A failing verify **must** exit `1` and (in human mode) print `✗ <id> NOT verified — do not trust this result.`
- [ ] A failing verify **must** record a failure on the recipe (`metrics.failures` increments, `metrics.last_failure_at` set).
- [ ] Verify **must** fail fast (exit `1`, no trust) if a `{{var}}` in the verifier command is unresolved, with message `verifier for "<id>" is missing inputs: <keys>`.
