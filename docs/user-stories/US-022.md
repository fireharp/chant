# US-022 — Relate enchantments: a v2 supersedes v1; a Go enchantment mirrors a Python one

> PLANNED — depends on typed `relations` and `spell_hash` identity in
> `docs/specs/enchantment-metadata.md` §3–4 and §7. Not yet implemented; testers
> should NOT run these criteria against the current CLI.

**As a** maintainer of a growing enchantment library
**I want** enchantments to carry typed relations (supersedes / mirrors / depends_on / implements)
**So that** lineage and equivalence are explicit and feed a unified coherence graph.

## Acceptance criteria (intended behavior)

- [ ] An enchantment **should** record `relations.supersedes: [<lineage_id>@<version>]` so a v2 is known to replace v1, and retrieval **should** prefer the superseding version.
- [ ] Two enchantments implementing the same procedure in different languages/packages (e.g. a Go and a Python revenue-by-channel) **should** be linkable via `relations.mirrors` and detectable as the same spell when they share a `spell_hash`.
- [ ] `spell_hash` **must** be derived from the normalized procedure (placeholders normalized, entrypoint canonicalized, sorted required columns) so the same procedure yields the same hash across repos.
- [ ] An enchantment **should** be able to declare `relations.depends_on: ["data:<schema>", "config:<file>"]` and `relations.implements: [US-0NN]` to tie it to data, config, and the story it fulfils.
- [ ] Relation kinds **must** be a subset of coherence's edge vocabulary (`supersedes`, `mirrors`, `depends_on`, `implements`, plus `specializes`/`generalizes`) so a repo running both tools gets one graph.
