# US-019 — Agent reuses a BitGN refund-approval enchantment instead of rewriting the solver

**As an** agent resolving a customer-service refund trial
**I want** chant to retrieve the `refund-chargeback-threat` enchantment and verify it
**So that** I reuse the tested refund-approval procedure rather than re-writing a deterministic solver each trial.

This is the canonical BitGN demo (t43–t44). The enchantment encodes: read
records -> security/ownership gate -> status-chain check -> side-effect -> answer
with refs. A new trial like "finalize refund for return ret_042" should route to
the refund family and be adapted to `ret_042`.

## Acceptance criteria

- [ ] Running `chant suggest --task "the customer wants a refund and is threatening a chargeback dispute" --json` **must** return `refund-chargeback-threat` as the top hit (refund task routes to the refund enchantment).
- [ ] The refund task **must not** route to a CSV/analytics enchantment (no cross-domain false hit).
- [ ] The hit `reasons` **should** show the lexical overlap reason (the refund enchantment carries no file/column signals, so it relies on task-text similarity).
- [ ] Reuse **must** be verifier-first: the refund enchantment's verifier (its fixture/grader case) **must** pass before the resolution is trusted; a suggest hit alone **must** carry `trusted: false`.
- [ ] When a refund enchantment that performs a side-effect is reused, `chant run` alone **must not** mark the result trusted — only a passing `chant verify` may.
- [ ] NOTE: this story depends on a captured `refund-chargeback-threat` enchantment with a real verifier and example input existing in the library (the bench `RET-003` synthetic recipe proves routing; a captured recipe is required to prove the e2e verify gate).
