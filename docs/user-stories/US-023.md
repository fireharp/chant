# US-023 — Verifier-first still holds for foreign hits

> PLANNED — depends on cross-package discovery (`--global` / `import`) from
> `docs/specs/enchantment-metadata.md` §2 and §6. Not yet implemented; testers
> should NOT run these criteria against the current CLI.

**As an** agent offered an enchantment imported from another project
**I want** that foreign hit to pass its verifier locally before I trust it
**So that** widening discovery never widens trust (no cross-package wrong-answer amplification).

## Acceptance criteria (intended behavior)

- [ ] A foreign enchantment surfaced by `chant suggest --global` **must** carry `trusted: false` and be treated as the most suspect kind of candidate.
- [ ] After `chant import <origin> <id>`, the agent **must** run `chant verify <id>` locally; only a passing local verifier **may** set `"trusted": true`.
- [ ] If the foreign enchantment's verifier fails in the local context (e.g. schema/runtime mismatch), the outcome **must** keep `"trusted": false` and recommend repair or invalidate — the foreign origin **must not** grant any trust shortcut.
- [ ] A foreign hit's `portability` contract (determinism, side_effects, input_contract, requires) **should** be shown so the agent can judge applicability before importing.
- [ ] Importing **must not** mutate the origin project; trust is established per-context, and a successful local verify **should** add the local context to the enchantment's `verified_in` evidence (feeding US-021 promotion).
