# US-020 — Discover a universal enchantment from another project

> PLANNED — depends on the cross-package registry and `--global`/`import`
> commands described in `docs/specs/enchantment-metadata.md` §6. Not yet
> implemented; testers should NOT run these criteria against the current CLI.

**As an** agent working in project B
**I want** `chant suggest --global` to surface a matching enchantment captured in project A
**So that** I reuse a portable, verified procedure across packages instead of re-solving it.

## Acceptance criteria (intended behavior)

- [ ] Running `chant suggest --task "compute revenue by channel" --global --json` **should** search the per-machine registry (`~/.chant/registry/index.json`) and return foreign hits.
- [ ] A foreign hit **should** be annotated with its `provenance.origin` (the repo it was born in) and its `scope` (`project`|`domain`|`universal`).
- [ ] A foreign hit's `reuse_command` **should** copy it locally first via `chant import <origin> <id>` (or by `spell_hash`) and then verify, rather than pointing straight at `chant verify`.
- [ ] Discovery **must** widen the candidate set only; a foreign hit **must** carry `trusted: false` until verified locally (see US-023).
- [ ] Local enchantments **should** be preferred over foreign ones at equal relevance, and `chant suggest --scope=domain+` **should** prefer portable (domain/universal) enchantments.
