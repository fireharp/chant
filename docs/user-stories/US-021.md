# US-021 — An enchantment graduates project → domain → universal as it is verified across repos

> PLANNED — depends on the scope/universality ladder and `verified_in` evidence
> tracking in `docs/specs/enchantment-metadata.md` §5. Not yet implemented;
> testers should NOT run these criteria against the current CLI.

**As a** chant maintainer with enchantments reused across many projects
**I want** an enchantment's `scope` to be promoted as its verifier passes in new contexts
**So that** universality is earned from evidence, not declared by hand.

## Acceptance criteria (intended behavior)

- [ ] A freshly captured enchantment **must** default to `scope: project`.
- [ ] Each passing verify in a new context **should** append a distinct entry to `verified_in` (`{ context, at }`); a repeat in the same context **must not** double-count.
- [ ] When `verified_in` holds ≥2 distinct contexts sharing a `domains` tag, the enchantment **should** be promotable to `scope: domain`.
- [ ] When `verified_in` holds ≥3 distinct contexts spanning ≥2 domains, the enchantment **should** be promotable to `scope: universal`.
- [ ] Promotion **must** be computed from `verified_in` evidence (e.g. via a future `chant promote <id>`), never set by hand in a way that bypasses evidence.
- [ ] Repeated verifier failures **should** demote scope and/or feed the existing stale/`invalidate` path (promotion is reversible; verifier-first governs both directions).
